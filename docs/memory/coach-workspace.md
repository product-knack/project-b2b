The odds-app native coach workspace (for `profiles.role = 'coach'`) was built end-to-end. There is exactly ONE coach: id `3d04b08d-f723-4d21-b2bc-8bc82ebcd2f5`, name "Coach", login `coach@oddsfitness.com` / `Coach@odds001` (62 trainers → 267 active clients live). `coach_trainers` is RLS-scoped to the coach — only readable under the coach's own session (CRM/trainer logins get 0 rows).

**Universal scope** (every coach page): `coach_trainers(coach_id=me) → trainer_clients(trainer_id IN …, actively_training=true) → clients`. Implemented as `coachScope(uid)` in [coachQueries.ts](odds-app/src/lib/coachQueries.ts) — 10 hooks: dashboard, clients, clients-overview, trainers, assessments, programs, progression, approved-plans, plans-overview, calendar. Screens in [coach.tsx](odds-app/src/screens/coach.tsx). SECURITY-DEFINER RPCs used: `get_pending_workout_plans_for_coach`, `get_pending_training_plans_for_coach`, `get_unique_client_plans` (all verified live).

**Role plumbing** (the pattern for adding any new role): `Role` type in [store.tsx](odds-app/src/store.tsx), `appRoleOf` in [auth.tsx](odds-app/src/auth.tsx), `coachNav` in [data.ts](odds-app/src/data.ts), drawer selection in [chrome.tsx](odds-app/src/components/chrome.tsx) line ~113, `SCREENS` map + post-login redirect in [Router.tsx](odds-app/src/Router.tsx), the duplicate redirect in `SignIn.doSignIn` ([trainer.tsx](odds-app/src/screens/trainer.tsx) ~72), and `ROLE_LABELS` in [navQueries.ts](odds-app/src/lib/navQueries.ts). There is NO route-level access control — role only picks the drawer nav + landing screen.

**GOTCHA — PostgREST caps any response at 1000 rows.** Raw-row aggregations (counting training_sessions / workout_plan_exercises) silently truncate; this coach has 14,450 sessions. Always page with `.range()` — see the `fetchAll()` helper in coachQueries.ts. Verify big-team counts before trusting them.

**Programs approve/reject/needs-revision** are wired as DIRECT table UPDATEs (no approval RPC exists): workout plans → `workout_plan_exercises` by `plan_id` (updates ALL exercise rows), training plans → `client_training_plans` by `id`. approve sets status='approved'+approved_by+approved_at; reject/needs_revision set status+coach_feedback (feedback required). Hook `useProcessCoachPlan` in coachQueries.ts. Coach UPDATE RLS on `workout_plan_exercises` VERIFIED live 2026-07-15 (no-op update returned all 10 exercise rows). **Programs page rebuilt 2026-07-15**: `useCoachProgramsV2` (key `['coach-programs','v2',uid]`) merges both pending RPCs newest-first (web parity) + processed `client_training_plans` w/ exact count + `coach_feedback`; `usePlanExercises(planId)` feeds a PlanViewerSheet (exercises grouped by body_part with sets/reps/RM%/load/tempo/rest/RIR/duration chips — port of web WorkoutPlanView). `file_path`/`document_url` are FULL public storage URLs → open with `Linking.openURL`. Pending-workout `plan_name` can be null — always fallback-title. liveSync maps workout_plan_exercises/client_training_plans → coach program keys. **Web Approved Plans page shows "34" but that's the 1000-row-cap bug** (its workout query doesn't paginate → only newest ~1000 exercise rows → ~30 plans + 4 docs); true totals live 2026-07-15: 1,277 unique approved workout plans (46,759 exercise rows) + 4 approved training docs. Native paginates fully and shows the real count — expect the user to notice the number mismatch vs web. Pending cards mirror web PlanHeaderCard: TRAINER name is the card title, subtitle "Workout/Training Plan for {client}".
